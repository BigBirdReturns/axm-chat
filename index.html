01 The Context Collapse

Vendor APIs are memory black holes.

You spend hours reasoning through architecture, strategic planning, and complex decisions with LLMs. That cognitive work is trapped inside fragmented chat interfaces controlled by vendors.

When you need to remember why a specific path was chosen or what alternatives were rejected, the context is gone. Search fails because the decisions were never formally committed.

The graveyard of failed attempts is the most valuable part of the record. It prevents the same dead ends later.

The Protocol
Not a search index — a provenance record.

The AXM protocol was built to verify autonomous robotics doctrine offline. axm-chat applies that same cryptographic rigidity to your personal knowledge base. Every decision is signed, every claim is traceable, every shard is yours.

03 Shard Anatomy

No black boxes. Just files.

A "shard" is not a proprietary database. It is a standard directory. You can zip it, email it, or query it with any data science tool. The security is in the cryptography, not the storage medium.

Click manifest.json to inspect the structure. This is the exact artifact axm-chat produces. axm-core mounts these directories directly into DuckDB at query time.

04 The Primitives
BLAKE3 Merkle Trees
Every claim and text block is hashed into a strict Merkle tree. Verify structural integrity of any shard entirely offline — no central server required.
ML-DSA-44 Signatures
Post-quantum signatures (FIPS 204) ensure that once a shard is compiled, the authorship and timestamp are mathematically immutable. Ed25519 also supported.
DuckDB + Parquet
Shards are stored locally as compressed Parquet files. At query time, the Spectra engine mounts them in-memory via DuckDB for instant relational analysis.
Lineage + References
When a decision shard supersedes a conversation shard, lineage@1 records the chain. references@1 links every decision claim back to the exact conversation turns that produced it.
Episodic Index
Every conversation gets a universal base pass: topic tags, named people, tools, project references, question text, resolution state, and tone — written to ext/[email protected].
Engineering Lens
A gated second pass for technical conversations. Extracts the problem statement, graveyard of failed attempts, working solution, and any architectural rule — written to ext/[email protected].
05 Local Execution
Install
pip install -e ./axm-genesis    # cryptographic compiler + verifier
pip install -e ./axm-core       # Spectra DuckDB query engine
pip install -e ./axm-chat       # chat spoke
Import
axm-chat import ./conversations.json    # one signed shard per conversation
Distill
ollama serve && ollama pull mistral
axm-chat distill                # episodic index + decision extraction
Query
axm-chat query "what decisions have we made"
axm-chat query "what failed before we solved the merkle problem"
axm-chat query "what changed since january"
axm-chat verify                 # offline Merkle + signature check

All data stays in ~/.axm/shards/ on your machine. Verify any shard entirely offline: axm-verify ./shard_dir

06 Three Layers

Memory has a shape.

AXM operates across three layers that share one container format but serve different temporal purposes. The protocol does not distinguish between them — all shards are structurally identical. The distinction is semantic, enforced by convention.

07 Definition of Correctness

The gold shard is the spec.

Not a test suite. Not a reference document. A specific directory of actual files — fm21-11-hemorrhage-v1/ — extracted from the US Army first aid field manual. It was compiled once. It will never be recompiled.

Any verifier that accepts this shard and rejects the invalid vectors in tests/vectors/ is conformant. A change that breaks the gold shard is rejected, full stop. This is what "frozen kernel" means in practice.

The gold shard is the definition of correctness. Not the code. Not the docs.

# axm-genesis conformance contract
# Any verifier that passes all of these is conformant.

shards/gold/fm21-11-hemorrhage-v1/    # MUST pass
tests/vectors/invalid/bad_signature   # MUST reject
tests/vectors/invalid/merkle_mismatch # MUST reject
tests/vectors/invalid/orphan_claim    # MUST reject
tests/vectors/invalid/null_in_column  # MUST reject

# Verify the gold shard yourself:
axm-verify shard shards/gold/fm21-11-hemorrhage-v1/ \
  --trusted-key keys/canonical_test_publisher.pub

# Output: {"status":"PASS","error_count":0,"errors":[]}
# Same output. Every machine. Every version. Forever.
08 Non-Selective Recording

REQ 5: you cannot delete frames.

For embodied spokes — robotics, sensor streams, any agent that maintains a binary hot buffer — the verifier enforces frame continuity. A gap in the cam_latents.bin sequence is a compliance failure. You cannot selectively delete the moment before an error occurred.

This is the technical implementation of the graveyard invariant: the record of what failed is as protected as the record of what succeeded. E_BUFFER_DISCONTINUITY fires on any gap, truncation, or bad magic — before the shard can be verified.

Chat shards and document shards are unaffected. This check runs only when content/cam_latents.bin is present.

# cam_latents.bin binary format
# File: [AXLF(4)] [records …]
# Record: [AXLR(4)] [ver(1)] [frame_id(4)] [len(4)] [payload]

# A gap between frame 4 and frame 6 → FAIL
frame 0  blake3:a1b2c3…  ✓
frame 1  blake3:d4e5f6…  ✓
frame 2  blake3:g7h8i9…  ✓
frame 3  blake3:j1k2l3…  ✓
frame 4  blake3:m4n5o6…  ✓
frame 5  [MISSING]       # E_BUFFER_DISCONTINUITY
frame 6  blake3:p7q8r9…  ✗ rejected

# axm-verify halts here. The shard cannot pass.
# The moment before the failure is part of the record.
